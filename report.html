3 · Eager release — memory freed on unmount

Nitro snaps back to baseline immediately; legacy drains slowly. On unmount nitro returned to the exact baseline in <1 s, while legacy stayed ~19 MB high and bled down over ~10 s. Neither leaks — but nitro frees eagerly & predictably, which matters when navigating away from a heavy screen. (Measured on live heap; footprint alone hides this because it doesn’t shrink promptly.)

4 · File load & dispose latency

Nitro exposes a programmatic file API — each run times RiveFileFactory.fromSource() (load) and file.dispose() (release). Legacy has no file API, so loading is measured as mount → first frame, and disposal is implicit.

5 · Property update overhead

Writing a data-bound number property in a tight loop (2000×). Nitro’s set() is a synchronous JSI call applied immediately; legacy’s setNumber() is an async bridge enqueue applied later on the native side.

Nitro Legacy
~17× faster per write — and that understates it. Nitro applies ~1,000,000 writes/sec synchronously; legacy enqueues ~57,000/sec and still has to process those bridge messages after the loop. This is where nitro’s architecture (direct JSI vs the async bridge) pays off: frequent property/state updates (sliders, gestures, data-driven animation) stay cheap and immediate.

Why nitro wins (the mechanism)

  • Shared RiveFile primitive — the file is parsed once and reused by every view, so both memory and mount time stay flat as views grow. Legacy parses per view.
  • Eager, deterministic disposal — native resources are freed on unmount via dispose(), not when the JS GC eventually runs.
  • Programmatic file API — load/dispose are explicit and fast; legacy only loads implicitly inside a view.

Method & caveats

  • Mount latency: time from mount to “ready” on every view — nitro via awaitViewReady(), legacy via the onPlay callback. Nitro pre-loads the shared file first, so the number reflects per-view instantiation.
  • Memory: in-app native module reads phys_footprint (Xcode’s gauge) and live malloc heap (malloc_zone_statistics). Heap reveals release; footprint does not shrink promptly. GPU/Metal texture memory is outside malloc and only shows in footprint/Instruments.
  • Simulator + debug build → absolute values are inflated and noisy; ratios are the signal. Device + Instruments (Allocations) numbers to follow.
